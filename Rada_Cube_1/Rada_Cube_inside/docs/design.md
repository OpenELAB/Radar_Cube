# FeedbackController 视听反馈设计

## 核心原则

1. 四颗 RGB 灯共同组成一个灯环，由一个 RGB task 统一驱动。
2. 普通灯效只由三个要素组成：颜色、动态效果和作用区域 `LedZone`。
3. 任意时刻只运行一种普通动态效果，不叠加左右两套不同的闪烁或呼吸效果。
4. 左右设备状态通过 `LedZone` 表达；系统级动画可以接管整个灯环。
5. 灯效发生变化时，根据最新系统状态重新生成输出，不恢复过期的像素画面。
6. 工作模式使用两个在线传感器距离的最小值，在线区域显示同一种距离灯效。
7. 距离反馈开始后不插入长语音，只允许短提示音短暂打断距离蜂鸣。
8. 短提示音结束后，不恢复旧的距离音频，而是根据最新距离重新选择蜂鸣档位。
9. 语音只用于模式最终结果、严重异常和需要用户采取行动的情况。
10. FeedbackController 负责视听策略和冲突仲裁；RGB 与 Speaker 驱动只负责执行。

## 旧设计存在的问题

旧设计以事件为中心，每个事件分别规定灯光、声音、阻塞和打断策略，容易产生以下问题：

- 左右局部灯效和四灯全局灯效互相覆盖。
- 多个状态同时发生时，不清楚应该叠加、打断还是恢复。
- 每增加一个事件，都要增加新的组合判断。
- 左右两侧可能出现不同动态效果，例如一侧闪烁、另一侧呼吸，用户难以理解。
- `queue`、`wait`、`interrupt` 和 `loop` 混合了音频类型、优先级与业务流程控制。
- 音频被打断后可能恢复已经过期的距离蜂鸣。

新设计统一灯环的视觉语言，通过作用区域表达左右设备差异，通过固定规则处理音频打断。

## 灯环模型

四颗 RGB 灯在外壳柔光板下形成一个灯环，逻辑上分为以下区域：

| LedZone | 含义 |
| --- | --- |
| `None` | 不显示普通区域灯效 |
| `Left` | 左边两颗灯 |
| `Right` | 右边两颗灯 |
| `Both` | 全部四颗灯 |

普通灯效由以下三个维度组成：

| 维度 | 示例 |
| --- | --- |
| 颜色 | 绿色、黄色、红色、蓝色、琥珀色 |
| 动态效果 | 常亮、闪烁、呼吸、脉冲 |
| 作用区域 | `None`、`Left`、`Right`、`Both` |

例如：

```text
绿色 + 慢闪 + Both
绿色 + 脉冲 + Left
红色 + 常亮 + Right
```

RGB 驱动只需要一个 task。每一帧统一计算闪烁相位、呼吸亮度或脉冲进度，再把结果写入 `LedZone` 指定的两颗或四颗灯，不需要为左右两侧分别创建 task。

## LedZone 状态管理

FeedbackController 维护当前可用的灯光区域：

| 系统阶段 | LedZone 表示的含义 |
| --- | --- |
| 配对模式 | 已完成配对的设备 |
| 唤醒阶段 | 已成功唤醒的设备 |
| 工作模式 | 当前在线并可提供距离数据的设备 |

典型状态转换：

```text
初始状态：None
左侧成功：Left
右侧成功：Right
左右均成功：Both
Both 状态下右侧掉线：Left
Both 状态下左侧掉线：Right
双侧掉线：None
```

`LedZone` 由 FeedbackController 根据最新设备状态更新。RGB 驱动不判断配对、唤醒、失联或业务模式。

## 统一灯语

建议使用稳定的颜色语义：

| 颜色 | 建议语义 |
| --- | --- |
| 蓝色 | 配置、配对、等待 |
| 绿色 | 正常、成功、可用 |
| 黄色或琥珀色 | 注意、降级、失联、低电 |
| 红色 | 危险、失败、系统不可用 |

建议使用稳定的动态语义：

| 动态效果 | 建议语义 |
| --- | --- |
| 顺时针旋转 | 启动、进入 |
| 逆时针旋转并渐暗 | 退出、关机、休眠 |
| 呼吸 | 等待连接或等待用户 |
| 单次脉冲 | 操作成功、设备上线 |
| 慢闪 | 一般状态提示 |
| 快闪 | 警告、危险或失败 |
| 常亮 | 稳定状态 |

颜色和动态语义应保持一致，不应在不同场景中表达相反含义。

## 配对模式灯效

### 开始配对

- 四灯蓝色呼吸。
- 当前已配对区域为 `None`。

### 第一侧配对成功

- 立即结束四灯蓝色呼吸。
- 成功侧绿色脉冲一次，然后绿色常亮。
- 尚未成功的一侧熄灭。
- 播放一次短促上升音。

### 第二侧配对成功

- 四灯显示绿色完成动画。
- 统一播放最终配对成功提示。
- 如果第二侧成功直接意味着配对流程完成，可以省略第二次单侧上升音，避免提示过密。

### 配对超时

- 已成功侧保持绿色。
- 未成功侧保持熄灭，或根据实机验证改为明确的失败颜色。
- 只播放一次最终配对结果，不连续播放多个左右失败语音。

## 唤醒阶段灯效

- 开始唤醒时，根据产品设计显示统一等待灯效。
- 单侧唤醒成功后，对应区域点亮，并把一个短促上升音加入唤醒反馈序列。
- 左右设备接近同时唤醒时，两个上升音仍按 ACK 到达顺序完整播放，不合并。
- 双侧唤醒成功后进入 `Both`。
- 两个上升音结束后，播放一次双侧唤醒成功语音。
- 业务流程等待整个唤醒反馈序列结束或超时，然后再开始距离视听反馈。
- 等待期间继续接收无线数据并缓存最新有效距离；进入距离反馈时直接使用最新值。

## 工作模式灯效

工作模式使用两个在线传感器距离的最小值，并把同一种距离灯效应用到当前在线区域。

建议映射如下：

| 距离状态 | 在线区域灯效 |
| --- | --- |
| 安全距离 | 绿色常亮 |
| 较远距离 | 绿色慢闪 |
| 中距离 | 黄色闪烁 |
| 近距离 | 红色快闪 |
| 危险距离 | 红色常亮 |

> 注意：以上颜色、距离等级和动态模式仅供设计讨论参考，可能与当前代码不一致。最终方案应根据实际产品设计、距离阈值、外壳柔光效果和实机用户测试确定。

单侧掉线时：

- 将掉线侧从在线 `LedZone` 中移除。
- 在线侧继续显示当前最小有效距离对应的统一灯效。
- 掉线侧默认熄灭。
- 播放一次短促下降音。
- 如果实机测试发现熄灭不够明确，可以将掉线侧改为固定故障颜色，但不引入左右两套不同的动态效果。

双侧掉线时：

- 停止距离灯效和距离蜂鸣。
- 进入系统不可用的全局灯效，不能只让四灯全部熄灭。
- 播放明确但简短的故障提示音。
- 如果设备仍处于工作模式，应根据产品测试决定是否周期性提醒系统不可用。

## 全局灯效

以下灯效始终控制全部四颗灯，不受当前 `LedZone` 限制：

- 上电顺时针跑马灯。
- 休眠或关机时逆时针跑马灯并渐暗。
- 配对最终完成动画。
- 恢复出厂动画。
- 双侧掉线或系统整体不可用。
- 严重系统故障。

全局灯效结束后，FeedbackController 根据最新业务状态重新生成普通灯效，不恢复全局灯效开始前保存的像素内容。

## 音频分类

音频只分为以下三类：

| 类型 | 用途 |
| --- | --- |
| 距离蜂鸣 | 持续表达当前最小距离 |
| 短提示音 | 上升音表示成功或恢复，下降音表示掉线或异常 |
| 最终语音 | 配对结果、配置结果、关机、严重故障或用户行动指导 |

灯光负责持续显示当前状态，短提示音负责提示状态变化，语音负责解释最终结果和必要操作。

## 音频打断规则

距离反馈开始后不播放长语音。短提示音可以短暂打断距离蜂鸣：

```text
暂停距离蜂鸣
播放短提示音
读取最新有效距离
根据最新距离重新开始对应蜂鸣
```

需要遵守以下规则：

1. 短提示音应尽量控制在 200 至 500 ms。
2. 提示结束后不得继续旧的距离蜂鸣档位。
3. 普通短提示音不能打断正在播放的完整语音，应等待、合并或丢弃。
4. 同一种掉线、恢复或低电事件不能反复播放，需要防抖和冷却时间。
5. 短时间内左右两侧同时发生相同异常时，应合并成一次双侧提示。
6. 一次性语音被严重故障取消后，不从中间位置继续；必要时从头重新播放。

## Wait 的使用范围

`wait` 不是一种音频类型，只表示业务流程是否需要等待最终提示结束。

声音在队列中等待前一个声音结束，不等于业务流程等待。除非事件明确要求流程等待，否则 `main` 应继续处理无线接收和业务状态。

适合等待的场景：

- 配对最终结果。
- 双侧唤醒成功后的完整反馈序列。
- 配置最终结果。
- 恢复出厂完成。
- 关机或进入深度睡眠。

不适合等待的场景：

- 工作模式的距离更新。
- 单侧掉线或恢复。
- 普通低电提示。
- 单侧配对或唤醒成功。

等待音频时只暂停业务状态转换，不能停止 ESP-NOW 接收、RGB task、Speaker task 或系统维护任务，并且必须设置最大超时时间。

## 必须处理的边界情况TODO

1. 失联必须持续达到阈值后才能确认，避免瞬时丢包触发下降音。
2. 恢复也需要连续收到有效数据后确认，避免上线和掉线反复切换。
3. 距离档位切换需要迟滞和最短保持时间。
4. 提示音播放期间状态再次改变时，只使用提示结束时的最新状态。
5. 过期的一次性事件应丢弃，不应形成不断增长的播放队列。
6. 双侧掉线后不能恢复距离蜂鸣，直到至少一侧重新提供有效数据。
7. 音频播放失败或超时不能阻塞业务流程。

## 模块职责

### main

- 负责配对、唤醒、工作、配置和休眠等业务流程。
- 把设备状态变化和业务事件通知给 FeedbackController。
- 不直接决定具体颜色、闪烁频率或音频打断策略。

### FeedbackController

- 维护当前 `LedZone` 和必要的视听状态。
- 根据业务状态选择颜色、动态效果和作用区域。
- 管理短提示音、距离蜂鸣和最终语音之间的冲突。
- 在提示结束后根据最新状态重新生成输出。

### RgbLedController

- 使用一个 task 驱动四颗 RGB 灯。
- 执行指定颜色、动态效果和 `LedZone`。
- 不理解配对、距离、失联或系统模式。

### SpeakerController

- 执行一次播放、循环播放和停止。
- 管理 WAV、I2S、音量和硬件资源。
- 不决定业务优先级或用户反馈策略。
